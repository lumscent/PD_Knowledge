# Формат отчёта — Design Audit

Используется при `/audit` для оформления результатов.

---

## Структура отчёта

### 1. Резюме
Короткий абзац: что проверялось, сколько проблем найдено, общее впечатление.

### 2. Сводная таблица проблем

| # | Экран | Проблема | Тип | Критичность | Решение |
|---|---|---|---|---|---|
| 1 | [Название экрана] | [Что не так] | Визуальное / UX / Копирайтинг / Тех долг | 🔴 Critical | [Что сделать] |
| 2 | ... | ... | ... | 🟠 High | ... |

### 3. Детальный разбор по типам

#### 🎨 Визуальные проблемы
Список с описанием, скриншот (если есть), решение.

#### 🧭 UX-проблемы
Список с описанием сломанного сценария, решение.

#### ✍️ Копирайтинг
Формат "было → стало" для каждой правки.

#### 🔧 Технический долг
Отклонения от дизайн-системы, несоответствия Figma vs продакшн.

### 4. Приоритизированный план правок

**Сделать сразу (Critical + High):**
1. ...

**Следующий спринт (Medium):**
1. ...

**Бэклог (Low):**
1. ...

### 5. Следующие шаги
- Создать задачи в Jira для Critical и High проблем
- Запланировать повторный аудит после правок

---

## Как подавать результаты

- Одна проблема = одна строка в таблице
- Не смешивай несколько проблем в одном пункте
- Для копирайтинга всегда показывай "было → стало"
- Если проблема в дизайн-системе — ссылайся на конкретный компонент в Storybook
