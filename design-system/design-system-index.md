# Design System Index

Статус: `draft`

---

## Назначение

Раздел `design-system/` хранит базовые правила визуального и интерфейсного проектирования для продуктовых дизайнеров.

Он нужен, чтобы:

- сохранять консистентность интерфейсов;
- быстрее проверять макеты;
- формировать единые правила для дизайн-ревью;
- помогать Claude учитывать дизайн-системные ограничения;
- готовить интерфейсы к передаче в разработку.

## Файлы раздела

| Файл | Назначение | Когда использовать |
|---|---|---|
| `visual-style.md` | Общий визуальный стиль интерфейсов | Когда нужно проверить визуальный тон, иерархию, плотность и акценты |
| `interface-rules.md` | Общие правила проектирования экранов и сценариев | Когда нужно создать, разобрать или проверить экран |
| `component-rules.md` | Правила использования компонентов | Когда задача связана с кнопками, инпутами, карточками, модалками, тултипами |
| `states.md` | Правила проектирования состояний интерфейса | Когда нужно описать default, loading, empty, error, success, disabled и другие состояния |

---

## Как дизайнеру использовать раздел

Перед созданием или ревью интерфейса дизайнер должен проверить:

1. Соответствует ли решение общим правилам интерфейса.
2. Есть ли понятная визуальная иерархия.
3. Правильно ли используются компоненты.
4. Учтены ли обязательные состояния.
5. Можно ли передать макет в разработку без устных объяснений.

---

## Как Claude должен использовать раздел

Claude должен обращаться к разделу `design-system/`, если задача связана с:

- созданием нового экрана;
- разбором UI;
- дизайн-ревью;
- UX-writing для интерфейса;
- компонентами;
- состояниями;
- визуальной консистентностью;
- подготовкой макета к разработке.

Если нужное правило отсутствует, Claude должен:

- явно указать, что правило пока не описано;
- не выдумывать стандарт команды;
- предложить, какой файл нужно дополнить.

---

## Минимальное правило

Любой экран должен быть проверен по трём уровням:

1. **Сценарий** — понятно ли, что пользователь делает и зачем.
2. **Интерфейс** — понятно ли, что доступно, что важно и что произойдёт дальше.
3. **Состояния** — предусмотрены ли loading, empty, error, success и disabled.
