# Product Index

Этот файл является индексом продуктов в репозитории `product-design-knowledge`.

Он помогает дизайнерам и AI-инструментам быстро понять:

- какие продукты описаны в репозитории;
- где находится продуктовый контекст;
- какие файлы читать для разных типов задач;
- какой статус заполнения у каждого продукта;
- каких данных пока не хватает.

---

## Как использовать этот файл

Перед началом работы с интерфейсной, UX- или продуктовой задачей нужно определить продукт.

Если продукт указан, перейти в соответствующую папку:

```text
products/[product-name]/
```

Если продукт не указан, использовать этот индекс, чтобы понять, какие продукты уже описаны в репозитории.

Если нужного продукта нет в индексе, его нужно добавить перед началом системной работы с задачами по этому продукту.

## MyLink

### Путь

```text
products/mylink/
```

### Статус

`draft`

Продуктовый контекст находится на этапе первичного заполнения. Пока не все данные могут быть добавлены или подтверждены.

Если Claude использует этот раздел, он должен явно отделять подтверждённые данные от предположений.

---

### Назначение раздела

Раздел `products/mylink/` используется для хранения продуктового и UX-контекста по MyLink.

Он нужен, чтобы дизайнеры и AI-инструменты могли учитывать:

- назначение продукта;
- цели продукта;
- пользовательские роли;
- ключевые сценарии;
- UX-принципы;
- известные проблемы;
- ограничения;
- контекст для интерфейсных задач.

---

## Какие файлы читать по MyLink

### Общий продуктовый контекст

Файл:

```text
products/mylink/product-card.md
```

Использовать, когда нужно понять:

- что такое MyLink;
- для чего нужен продукт;
- какие бизнес-цели он поддерживает;
- какие пользователи с ним работают;
- какие ограничения важно учитывать.

---

### Пользователи и роли

Файл:

```text
products/mylink/users-and-roles.md
```

Использовать, когда задача связана с:

- пользовательскими ролями;
- правами доступа;
- внутренними и внешними пользователями;
- различиями между сценариями разных ролей;
- ограничениями в интерфейсе для разных пользователей.

---

### Ключевые сценарии

Файл:

```text
products/mylink/key-flows.md
```

Использовать, когда задача связана с:

- пользовательским флоу;
- переходами между экранами;
- статусами;
- действиями пользователя;
- ошибками;
- edge cases;
- альтернативными сценариями.

---

### UX-принципы продукта

Файл:

```text
products/mylink/ux-principles.md
```

Использовать, когда нужно:

- проверить интерфейсное решение;
- выбрать UX-подход;
- оценить понятность сценария;
- проверить, не создаёт ли интерфейс лишнюю когнитивную нагрузку;
- убедиться, что ограничения и статусы объяснены пользователю.

---

### Известные проблемы

Файл:

```text
products/mylink/known-problems.md
```

Использовать, когда задача связана с:

- улучшением существующего интерфейса;
- известными болями пользователей;
- повторяющимися проблемами в сценариях;
- UX-долгом;
- проблемами, которые уже были замечены командой.

---

### AI-контекст продукта

Файл:

```text
products/mylink/ai-context.md
```

Использовать, когда Claude или другой AI-инструмент работает с задачей по MyLink.

Файл должен помогать AI понять:

- какие данные учитывать в первую очередь;
- какие разделы продукта читать;
- какие ограничения не выдумывать;
- как формировать рекомендации по интерфейсу.

---

## Как Claude должен использовать этот индекс

Если пользователь указывает продукт `MyLink`, Claude должен:

1. Перейти в папку `products/mylink/`.
2. Прочитать `product-card.md`.
3. Проверить `users-and-roles.md`, если задача связана с ролями.
4. Проверить `key-flows.md`, если задача связана со сценарием.
5. Проверить `ux-principles.md`, если задача связана с UX-решением.
6. Проверить `known-problems.md`, если задача связана с улучшением текущего интерфейса.
7. Проверить `ai-context.md`, если нужно сформировать рекомендацию с учётом продуктового контекста.

Если нужный файл отсутствует или не заполнен, Claude должен указать это в ответе.

---

## Что нельзя делать

Claude не должен:

- выдумывать данные о продукте;
- считать пустой или черновой файл подтверждённым источником;
- использовать предположения как факты;
- ссылаться на файл как на источник, если в нём нет нужной информации;
- игнорировать статус `draft`, `needs-review` или `deprecated`.

---

## Статусы продуктов

| Статус | Значение | Можно ли использовать как источник |
|---|---|---|
| `draft` | Черновик, данные ещё заполняются | Можно использовать осторожно, с пометкой |
| `active` | Актуальный продуктовый контекст | Да |
| `needs-review` | Требует проверки или обновления | Только с пометкой |
| `deprecated` | Устаревший продукт или раздел | Нет, только как исторический контекст |

---

## Как добавить новый продукт

Чтобы добавить новый продукт, нужно:

1. Создать папку в `products/`.
2. Назвать папку коротко и понятно, латиницей, в нижнем регистре.

Пример:

```text
products/new-product-name/
```

3. Добавить базовые файлы:

```text
product-card.md
users-and-roles.md
key-flows.md
ux-principles.md
known-problems.md
ai-context.md
```

4. Добавить продукт в таблицу “Список продуктов” в этом файле.
5. Указать статус продукта.
6. Добавить краткое назначение продукта.
7. Проверить, что Claude может понять, какие файлы читать для задач по этому продукту.
