# MyLink UX Principles

Статус: `draft`  
Продукт: `MyLink`

---

## Назначение

Файл описывает UX-принципы для проектирования интерфейсов MyLink.

Используется, чтобы:
- проверять интерфейсные решения;
- сохранять консистентность сценариев;
- снижать барьеры для отклика;
- делать путь кандидата понятным;
- помогать Claude учитывать продуктовую UX-логику.

---

## Основные принципы

| Принцип | Что означает | Где применять |
|---|---|---|
| Быстро объяснять ценность сервиса | Пользователь должен быстро понять, что MyLink помогает найти работу и перейти к подходящей вакансии | Главная страница, первый экран |
| Делать поиск вакансий основным сценарием | Поиск и переход к вакансиям должны быть заметными и простыми | Главная, список вакансий, фильтры |
| Показывать ключевые условия до отклика | Кандидат должен видеть важные параметры вакансии до принятия решения | Карточка вакансии, детальная страница |
| Снижать барьер отклика | Отклик должен требовать минимум действий и понятный набор данных | Форма отклика, CTA, регистрация |
| Объяснять следующий шаг | После отклика пользователь должен понимать, что произошло и чего ждать дальше | Success state, уведомления, форма отклика |
| Не скрывать причины ограничений | Если действие недоступно, интерфейс должен объяснить почему | Disabled states, формы, авторизация |
| Помогать при пустых результатах | Empty state должен помогать продолжить поиск, а не просто сообщать, что ничего нет | Поиск, фильтры, список вакансий |
| Работать с доверием | Пользователь должен понимать, кто получит его данные и зачем они нужны | Форма отклика, регистрация, согласия |
| Учитывать мобильный сценарий | Поиск работы часто происходит с телефона, поэтому ключевые действия должны быть удобны на мобильном | Все основные сценарии |
| Разделять факт и предположение | Если данные неполные или требуют проверки, интерфейс не должен делать категоричных выводов | Статусы, подсказки, системные сообщения |

---

## Принципы для вакансий

В карточке или деталях вакансии важно показывать:

- название вакансии;
- работодателя;
- город или локацию;
- зарплату, если доступна;
- график;
- формат работы;
- ключевые требования;
- основные обязанности;
- условия;
- CTA для отклика.

Если часть данных отсутствует, интерфейс должен не ломаться, а показывать понятное состояние.

---

## Принципы для отклика

Отклик должен быть:

- заметным;
- простым;
- понятным;
- безопасным для пользователя;
- с ясным результатом после отправки.

Перед отправкой данных кандидат должен понимать:

- какие данные он передаёт;
- кому они будут отправлены;
- что произойдёт после отклика;
- нужно ли ждать обратную связь;
- можно ли будет изменить данные позже.

---

## Принципы для форм

Формы должны:

- запрашивать только необходимые данные;
- явно показывать обязательные поля;
- объяснять, зачем нужны чувствительные данные;
- сохранять введённую информацию при ошибке;
- показывать понятные ошибки;
- быть удобными на мобильных устройствах.

---

## Принципы для состояний

Для ключевых сценариев нужно предусматривать состояния:

- default;
- loading;
- empty;
- error;
- success;
- disabled;
- no results;
- unavailable;
- permission required.

Каждое состояние должно отвечать на три вопроса:

1. Что произошло?
2. Почему это произошло?
3. Что пользователь может сделать дальше?

---

## Как Claude должен использовать файл

Claude должен использовать эти принципы при:

- разборе экранов MyLink;
- постановке интерфейсных задач;
- UX-writing;
- проектировании сценариев;
- проверке карточек вакансий;
- проверке формы отклика;
- анализе empty/error/disabled states.

Если рекомендация противоречит этим принципам, Claude должен явно указать конфликт и объяснить причину.
